# /dev:discuss

架构讨论模式（只读）。产品经理主持，架构师做技术顾问，你做决策。不修改任何代码。

## 用法

```bash
/dev:discuss <讨论主题>
```

## 参数

| 参数 | 说明 |
|------|------|
| `讨论主题` | 要讨论的技术主题，如 "用户权限系统怎么设计" |

## 适用场景

- 不确定怎么实现，想讨论方案
- 技术选型犹豫，需要分析利弊
- 架构方向有争议，需要多方视角
- 想在写代码前对齐思路

## 不适用

- 已经知道怎么做 -- 用 `/dev:feat` 或 `/dev:fix`
- 想审查代码质量 -- 用 `/dev:review`
- 想排查 bug -- 用 `/dev:investigate`

## 执行流程

```
/dev:discuss 用户权限系统怎么设计
    |
Step 1: 解析讨论主题
    |
Step 2: 产品经理开场 (opus)
|        澄清问题，收集上下文（最多 3 轮提问）
    |
Step 3: 架构师分析 + 多轮讨论 (opus)
|        分析 2-3 个可行方案，讨论利弊（最多 3 轮）
    |
Step 4: 生成会议纪要 -> .dev/discuss/MINUTES.md
    |
Step 5: Recorder (sonnet) -- 记录决策和原因
    |
Done. 建议具体开发模式
```

不使用 git、不创建分支。

## 产物文件

| 文件 | 说明 |
|------|------|
| MINUTES.md | 会议纪要（讨论要点、决策汇总、行动项） |

## 讨论流程

1. **产品经理** -- 了解背景、澄清核心问题、确认成功标准
2. **架构师** -- 分析 2-3 个可行方案（含利弊）、给出推荐
3. **用户** -- 做决策，选择方案或提出追问

## 常见后续命令

讨论完成后，根据决策建议：

| 决策 | 建议下一步 |
|------|-----------|
| 实现新功能 | `/dev:feat <功能描述>` |
| 重构现有代码 | `/dev:refactor <重构描述>` |
| 修复问题 | `/dev:fix <问题描述>` |

## 相关命令

- [/dev:review](./review) -- 代码审查
- [/dev:feat](./feat) -- 新功能开发
- [/dev:refactor](./refactor) -- 代码重构
