# Dev 命令

多 Agent 协作开发工作流命令集，自动化完成从需求讨论到验收的完整开发周期。

## 工作流概览

```
/dev:start <功能描述>
    ↓
1. 创建分支 (feat/fix/refactor)
    ↓
2. 产品讨论 → Agent 与用户讨论需求、绘制线框图（唯一人类介入点）
    ↓
3. 规划 → Agent 基于 PRD 生成任务分解
    ↓
4. 测试设计 → Agent 生成测试用例文档（先想清楚要测什么）
    ↓
5. 开发循环（逐任务）：
   Developer Agent 开发 → 验证 → 下一个任务
    ↓
6. 验收 → 生成验收说明书
    ↓
用户自行 /git:finish 合并分支
```

## Agent 角色

| Agent | 职责 | 模型 |
|-------|------|------|
| dev-product | 产品经理 — 需求讨论、线框图、PRD | opus |
| dev-planner | 规划 — 任务分解、依赖分析 | opus |
| dev-developer | 开发 — 逐任务编码实现 | sonnet |
| dev-tester | 测试设计 — 测试用例文档（先设计，后开发） | sonnet |
| dev-recorder | 记录 — 项目经验沉淀 | sonnet |

## 核心原则

1. **调度器模式** — 主命令只做调度，所有技术工作通过 Agent 委托
2. **产品先行** — 先讨论需求再动手，产品讨论是唯一人类介入点
3. **先设计再开发** — 规划完成后生成测试用例文档，明确"要验证什么"再写代码
4. **逐任务循环** — 开发→验证→决策是一个循环，每轮只做 1 个任务
5. **文件驱动** — 所有状态写入 `.dev/plan/<branch>/` 目录，不依赖上下文记忆

## 工作流架构

```
┌─────────────────────────────────────────────┐
│  Step 1-3: 解析 → 创建分支 → 初始化          │
└──────────────────┬──────────────────────────┘
                   v
         ┌─────────────────┐
         │  Step 4: 产品讨论│  一次性，与用户交互
         │  spawn product  │  输出 PRD + 线框图
         └────────┬────────┘  (唯一人类介入点)
                  v
         ┌─────────────────┐
         │  Step 5: 规划   │  一次性，基于 PRD
         │  spawn planner  │
         └────────┬────────┘
                  v
         ┌─────────────────┐
         │  Step 6: 测试设计│  一次性，基于 PLAN.md
         │  spawn tester   │  用户故事 + 测试用例文档
         └────────┬────────┘
                  v
         ┌─────────────────┐
         │  Step 7: 开发   │  每次只做 1 个任务
         │  spawn developer│
         └────────┬────────┘
                  v
         ┌─────────────────┐
         │  Step 8: 验证   │  只验证当前任务
         │  执行验证命令    │── 失败 ──> spawn developer 修复 ──┐
         └────────┬────────┘                                    │
                  v                                              │
         ┌─────────────────┐                                    │
         │  Step 9: 决策   │──────── 还有任务 ─────────────────┘
         │  还有任务?      │  → 回到 Step 7 开发下一个任务
         └────────┬────────┘
                  v (全部完成)
         ┌─────────────────┐
         │  Step 10: 验收  │
         │  生成验收说明书  │
         └─────────────────┘
```

## 分支类型

| 关键词 | 分支类型 |
|--------|---------|
| 修复、bug、fix、错误 | `fix/<slug>` |
| 重构、refactor、优化、清理 | `refactor/<slug>` |
| 默认 | `feat/<slug>` |

## 状态文件

工作流状态存储在 `.dev/plan/<branch-name>/` 目录：

| 文件 | 用途 |
|------|------|
| `PRD.md` | 产品需求文档（产品讨论产出） |
| `PLAN.md` | 任务分解计划（规划产出） |
| `TEST-DESIGN.md` | 测试用例设计文档（测试设计产出） |
| `TASK-LOG.md` | 执行记录（逐任务跟踪） |
| `ACCEPTANCE.md` | 验收说明书（全部完成后的产出） |
