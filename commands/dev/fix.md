# /dev:fix — Bug 修复模式

自包含的 bug 修复工作流：架构师诊断根因，开发者实施修复。独立运行，不依赖 `/dev:start`。

## 适用场景

- 已知 bug，需要诊断根因并修复
- 线上问题快速响应（非 hotfix 级别）
- 用户反馈的缺陷，需要分析+修复
- 需要比 `/dev:patch` 更严格的流程（文档、验证、验收）

## 不适用

- 一行修改、配置调整（用 `/dev:patch`）
- 紧急线上 hotfix（用 `/dev:start` + hotfix 关键词）
- 大规模重构（用 `/dev:start --standard`）

## 参数

- `--git`: 创建 fix/* 分支（默认行为）
- `--no-git`: 不创建分支，在当前分支上直接修复
- `--git` 和 `--no-git` 互斥，同时指定时报错
- `问题描述`: bug 的症状描述，支持中文。如 "登录后页面白屏"

## 绝对禁止 -- 你是调度器，不是执行者

你只做调度和文件状态管理。**禁止执行以下操作**：

- 读取 src/ 下的源代码文件
- 分析代码逻辑、bug 原因、技术方案
- 编写或修改任何源代码
- 运行测试、构建、安装依赖
- 做任何技术判断（根因分析、修复方案选择）

**唯一允许的操作：**

- 读取工作流状态文件（PRD.md, ACCEPTANCE.md）
- 读取 `.dev/config.yml` 项目配置
- 执行 git 命令（创建分支、查看状态、获取 diff/log）
- 使用 Agent() spawn 各阶段 Agent，通过返回值获取结果
- 写入工作流状态文件（ACCEPTANCE.md）
- 写入 `.dev/config.yml`（Step 3 初始化）

**如果你发现自己正在阅读源代码或思考技术方案 -> 立即停止 -> 改用 Agent 工具委托。**

## 流程总览

```
用户输入: /dev:fix 登录后页面白屏，控制台报 TypeError
    |
Step 1: 解析参数 -> use_git=true/false, bug_description
    |
Step 2: 创建分支 (if use_git) -> fix/<slug> from develop
    |
Step 3: 初始化状态 -> .dev/plan/fix-<slug>/ + .dev/config.yml
    |
Step 4: 架构师诊断 (inline, opus) -> PRD.md (bug report format)
    |
Step 5: 用户确认 -> AskUserQuestion
    |
Step 6: 开发者修复 (inline, sonnet) -> 实施修复
    |
Step 7: 验证 -> build/lint/test
    |
Step 8: 记录 (inline, sonnet) -> 更新 TASK-LOG
    |
Step 9: 验收 -> ACCEPTANCE.md
    |
Step 10: 最终摘要
```

总耗时预期：5-10 分钟（2 次主要 Agent spawn + 1 次确认 + 验证）。

## 执行步骤

### Step 1: 解析输入

解析 `$ARGUMENTS` 获取标志和 bug 描述。

**标志解析：**

```
1. 扫描参数中的 --git 和 --no-git 标志
2. 如果同时存在 --git 和 --no-git，报错退出
3. 设置变量：
   - --git      -> use_git = true
   - --no-git   -> use_git = false
   - 未指定     -> use_git = true  （fix 模式默认创建分支）
4. 从剩余文本中提取 bug_description
5. 如果没有 bug_description，用 AskUserQuestion 询问用户要修复什么 bug
```

**生成 slug：** 从 bug 描述中提取关键词，生成短横线连接的 slug（如 "登录白屏" -> "login-blank-screen"）。

**如果 use_git=false，slug 仍用于 `.dev/plan/fix-<slug>/` 目录命名。**

---

### Step 2: 创建分支（仅 use_git=true）

**跳过条件：** use_git=false 时直接跳过此步骤，在当前分支上工作。

根据 bug 描述生成 slug，从 develop 创建 `fix/<slug>` 分支：

```bash
# 检测 develop 分支是否存在
git rev-parse --verify develop >/dev/null 2>&1
if [ $? -ne 0 ]; then
  # develop 不存在，尝试 main/master
  if git rev-parse --verify main >/dev/null 2>&1; then
    BASE_BRANCH=main
  elif git rev-parse --verify master >/dev/null 2>&1; then
    BASE_BRANCH=master
  else
    BASE_BRANCH=$(git branch --show-current)
  fi
else
  BASE_BRANCH=develop
fi

# 检查工作区是否干净，不干净则 stash
STASH_NEEDED=false
if ! git diff --quiet || ! git diff --cached --quiet || [ -n "$(git ls-files --others --exclude-standard)" ]; then
  STASH_NEEDED=true
  git stash push -m "dev-fix-auto-stash" --include-untracked
fi

# 切换到 base 分支并拉取最新
git checkout "$BASE_BRANCH"
git pull origin "$BASE_BRANCH" 2>/dev/null || true

# 检查分支是否已存在
if git show-ref --verify --quiet "refs/heads/fix/$SLUG"; then
  # 分支已存在，切换到它
  git checkout "fix/$SLUG"
else
  git checkout -b "fix/$SLUG"
fi

# 恢复 stash
if [ "$STASH_NEEDED" = true ]; then
  git stash pop 2>/dev/null || true
fi
```

**记录：** `BASE_BRANCH` 和 `CURRENT_BRANCH="fix/$SLUG"` 用于后续步骤。

---

### Step 3: 初始化状态目录和项目配置

创建工作流状态目录：

```bash
mkdir -p .dev/plan/fix-$SLUG
```

**创建 `.dev/config.yml`：**

```yaml
# Auto-generated by /dev:fix workflow. Do not edit manually.

git:
  base_branch: <BASE_BRANCH>       # develop or main/master
  current_branch: <CURRENT_BRANCH> # fix/<slug> or current branch name
  slug: <SLUG>
  branch_type: fix

workflow:
  mode: fix
  use_git: <use_git>               # true or false

project:
  type: null
  language: null
  framework: null
  ui_library: null
  bundler: null
  runtime: null
  dependency_manager: null
  monorepo: false

source_layout:
  root: null
  entry_points: []
  directories: []

build:
  command: null
  type_check_command: null
  lint_command: null
  dev_command: null

test:
  framework: null
  unit_command: null
  single_test_command: null
  config_file: null

verification:
  commands: []

conventions:
  commit_format: null
```

**创建 `.dev/plan/fix-$SLUG/TASK-LOG.md`：**

```markdown
# Task Log: fix/<slug>

| Step | Agent | Status | Summary |
|------|-------|--------|---------|
| 诊断 | dev-architect | pending | - |
| 修复 | dev-developer | pending | - |
| 记录 | dev-recorder | pending | - |
```

---

### Step 4: 架构师诊断

以内联方式 spawn dev-architect，专注于 bug 诊断而非架构设计：

```
Agent(
  subagent_type="dev-architect",
  model="opus",
  prompt="
    <bug_description><用户输入的 bug 描述></bug_description>
    <project_root><项目根目录绝对路径></project_root>
    <config_path>.dev/config.yml</config_path>
    <prd_path>.dev/plan/fix-<SLUG>/PRD.md</prd_path>

    你是架构师，当前任务是 **bug 诊断**（不是架构设计）。

    请执行以下分析：

    1. **复现路径**: 根据描述，推断 bug 的触发条件和复现步骤
    2. **根因分析**: 阅读相关源代码，定位 bug 的根本原因
    3. **影响范围**: 列出所有受影响的文件和模块
    4. **修复方案**: 提出具体的修复方法，每个文件需要改什么、为什么
    5. **风险评估**: 修复可能引入的副作用，需要回归测试的范围

    将诊断结果写入 <prd_path>，格式为 Bug Report（不是产品需求文档）：

    ```markdown
    # Bug Report: <bug 简述>

    ## 症状
    <用户描述的 bug 现象>

    ## 根因
    <根本原因分析>

    ## 受影响文件
    - <file1>: <为什么受影响>
    - <file2>: <为什么受影响>

    ## 修复方案
    - <file1>: <具体改动>
    - <file2>: <具体改动>

    ## 风险点
    - <风险1>
    - <风险2>
    ```

    同时更新 <config_path> 中的 project、source_layout、build、test、verification、conventions 段（根据项目实际情况自动发现填充）。

    完成后返回简要摘要：根因（1 句话）+ 受影响文件数 + 修复方案概述。
  "
)
```

**Agent 返回后：**

1. 确认 `.dev/plan/fix-$SLUG/PRD.md` 已生成
2. 确认 `.dev/config.yml` 的项目信息段已填充
3. 提取关键信息用于 Step 5 确认：根因摘要、受影响文件列表、修复方案概述
4. 更新 TASK-LOG.md 诊断行状态为 done

---

### Step 5: 用户确认

将架构师的诊断摘要展示给用户：

```
AskUserQuestion(questions=[{
  header: "诊断确认",
  multi_select: false,
  options: [
    { label: "确认修复", description: "按诊断方案实施修复" },
    { label: "调整方案", description: "我想补充信息或修改修复方案" },
    { label: "取消", description: "不修复了" }
  ],
  question: "架构师诊断结果：

  **根因**: <根因摘要>
  **受影响文件**: <文件列表>
  **修复方案**: <方案概述>
  **风险**: <风险摘要>

  确认按此方案修复？"
}])
```

- **确认修复** -> 进入 Step 6
- **调整方案** -> 用户补充说明后，重新 spawn architect（回到 Step 4）或直接调整 prompt 进入 Step 6
- **取消** -> 显示取消信息，结束工作流

---

### Step 6: 开发者修复

根据确认的诊断方案 spawn 开发者：

```
Agent(
  subagent_type="dev-developer",
  model="sonnet",
  prompt="
    <project_root><项目根目录绝对路径></project_root>
    <config_path>.dev/config.yml</config_path>
    <prd_path>.dev/plan/fix-$SLUG/PRD.md</prd_path>

    你是开发者。当前任务是 **bug 修复**。

    根据 PRD.md 中的诊断报告和修复方案，实施代码修复。

    要求：
    - 严格按照 PRD.md 中的修复方案修改文件
    - 修复应该是最小化的：只改必要的代码，不做额外重构
    - 如果修复过程中发现新问题或方案需要调整，在返回摘要中说明
    - 实现完成后提交代码（commit message 格式: fix(<scope>): <简洁描述>）

    <user_adjustments>
    <用户调整内容，如无则为空>
    </user_adjustments>
  "
)
```

**Agent 返回后：**

1. 更新 TASK-LOG.md 修复行状态为 done，填入摘要
2. 记录 developer 返回的关键信息：实际修改了哪些文件、是否有方案调整

---

### Step 7: 验证

读取 `.dev/config.yml` 的 `verification.commands`。如果为空，依次检查 `build.command`、`lint_command`、`test.unit_command`。

**如果有验证命令，逐个执行：**

```bash
# 从 config.yml 读取验证命令并执行
# 如果 verification.commands 非空，执行每个命令
# 如果为空，尝试执行 build + lint + test（如果配置了的话）
```

**验证结果处理：**

| 结果 | 动作 |
|------|------|
| 全部通过 | 进入 Step 8 |
| 有失败 | 将失败输出传回 developer agent，要求修复验证问题（最多重试 2 轮） |
| 重试 2 轮仍失败 | 记录失败状态，进入 Step 8，在 ACCEPTANCE.md 中标注 |

**如果没有配置任何验证命令，跳过此步骤。**

---

### Step 8: 记录

以内联方式 spawn dev-recorder，传入实际的变更信息：

**收集 recorder 需要的数据：**

```bash
# 获取变更文件列表
CHANGED_FILES=$(git diff --name-only $BASE_BRANCH..HEAD 2>/dev/null || git diff --name-only HEAD~1..HEAD)

# 获取 commit hash
COMMIT_HASH=$(git log --oneline -1 --format="%h")

# 获取完整 commit 信息
COMMIT_LOG=$(git log --oneline $BASE_BRANCH..HEAD 2>/dev/null || git log --oneline -5)
```

```
Agent(
  subagent_type="dev-recorder",
  model="sonnet",
  prompt="
    <project_root><项目根目录绝对路径></project_root>
    <config_path>.dev/config.yml</config_path>
    <task_log_path>.dev/plan/fix-$SLUG/TASK-LOG.md</task_log_path>

    你是经验记录员。记录本次 bug 修复的完整信息。

    **修复信息：**
    - Bug 描述: <bug_description>
    - 根因: <从 PRD.md 提取的根因摘要>
    - 修复方案: <从 PRD.md 提取的修复方案摘要>
    - 变更文件: <CHANGED_FILES>
    - Commit: <COMMIT_HASH>
    - Commit 日志: <COMMIT_LOG>
    - 验证结果: <通过/失败/跳过>

    更新 TASK-LOG.md，填写所有步骤的状态和摘要。

    完成后返回简要记录摘要。
  "
)
```

**Agent 返回后：** 确认 TASK-LOG.md 已更新。

---

### Step 9: 验收

生成验收说明书 `.dev/plan/fix-$SLUG/ACCEPTANCE.md`。

**从 PRD.md、TASK-LOG.md 和 git log 提取信息，生成验收文档：**

```markdown
# 验收说明书: fix/<slug>

## Bug 症状

<从 PRD.md 提取的 bug 现象描述>

## 根因分析

<从 PRD.md 提取的根因>

## 修复内容

<从 PRD.md 和 TASK-LOG.md 提取的修复方案和实际改动>

## 变更文件

<CHANGED_FILES 列表>

## 验证结果

| 验证项 | 结果 |
|--------|------|
| build | <通过/失败/跳过> |
| lint | <通过/失败/跳过> |
| test | <通过/失败/跳过> |

## 提交记录

<COMMIT_LOG>

## 风险提示

<从 PRD.md 提取的风险点，以及实际验证中是否覆盖>
```

---

### Step 10: 最终摘要

显示工作流完成摘要：

```
================================================================
 FIX WORKFLOW COMPLETE
================================================================
 Bug: <bug 描述>
 分支: <fix/<slug> 或 当前分支名>
 根因: <根因摘要（1 句话）>
 修复: <修复方案摘要（1-2 句话）>
 变更: <N 个文件>
 验证: <通过/失败/跳过>
 提交: <commit hash 前 7 位>
 验收说明书: .dev/plan/fix-<slug>/ACCEPTANCE.md

 下一步:
   查看验收说明书 -- 确认修复是否符合预期
   /dev:status     -- 查看工作流状态
   /git:finish     -- 合并到 <base_branch> （如果 use_git=true）
================================================================
```

**如果 use_git=true：** 额外提示 `使用 /git:finish 完成分支合并。`

**如果 use_git=false：** 提示 `代码已提交到当前分支。请手动检查变更。`

## 与其他命令的对比

| | `/dev:patch` | `/dev:fix` | `/dev:start --quick` |
|---|---|---|---|
| Git 分支 | 否 | 可选（默认是） | 是 |
| 文档生成 | 否 | Bug Report + ACCEPTANCE | PRD + ACCEPTANCE |
| Agent 数 | 2 (inline) | 3 (inline) | 2 (inline) |
| 人类确认 | 1 次 | 1 次（诊断后） | 1 次（产品讨论） |
| 根因分析 | 无 | 有 | 无 |
| 验收文档 | 无 | 有 | 有 |
| 适用场景 | typo/一行修复 | 需要诊断的 bug | 小功能开发 |
| 预期耗时 | 2-3 min | 5-10 min | 5-8 min |

## 错误处理

### 分支创建失败

```
如果 git checkout 或 git checkout -b 失败：
1. 检查是否有未提交的冲突
2. 向用户报告错误
3. 建议使用 --no-git 模式，或手动解决 git 问题后重试
```

### 架构师诊断失败

```
如果 architect agent 无法定位根因：
1. 向用户报告诊断未能确定根因
2. 建议用户提供更多信息（复现步骤、错误日志、截图等）
3. 用户补充信息后，重新 spawn architect
```

### 开发者修复失败

```
如果 developer agent 修复失败：
1. 检查是否是诊断方案有误
2. 可以重新 spawn architect 进行二次诊断
3. 向用户报告失败原因，请求人工介入
```

### 验证持续失败

```
如果验证重试 2 轮仍失败：
1. 在 ACCEPTANCE.md 中标注验证失败
2. 向用户报告失败详情
3. 建议手动验证或调整修复方案
```
